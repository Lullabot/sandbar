brainstorm · not a decision

~ Five ways the sand TUI could feel crafted

sand manages disposable Lima VMs for Claude Code. Today it's one full-screen at a time. These five concepts — a dense dashboard, a palette room, a modal bulk-editor, a spatial drill, and a card board — each rethink the whole experience, and each ships a wide and a narrow mockup because responsiveness runs through all of them.

The spark was FlameConnect: a persistent status band, twin bordered panes, a docked timestamped log, a footer command bar, and a ^p palette. That crafted feel is the target.

the baseline these push against

Three things about sand today

Every concept below is trying to fix the same three frictions in the current single-screen app.

Screen-swap navigation

Reading one VM means leaving the list. You can never scan the fleet and inspect a VM at the same time.

Provisioning takes the screen hostage

beginStream flips to a full-screen log and freezes every key for minutes — no queueing a second VM.

Keybinding pressure & drift

d is delete and download, s is start and secrets. The keymap and help are hand-kept in parallel and can disagree.

build once, then A/B the shells

Four of the five want the same engine

The most useful finding: build these three foundations once and you can prototype several of the concepts cheaply on top. Add a responsive classify(w,h) and idle-gating as cross-cutting discipline.

Action registry

One Command{name, keys, help, enabledFor(vm), run} list that the keymap, footer, which-key and palette all render from. Kills the defaultKeys()/viewHelp() drift; enabledFor gives you state-gated verbs for free.

Concurrent job registry

Generalize today's single beginStream reader/output/cancel into a map[vmName]job, so several provisions and transfers stream at once while the UI stays live. The load-bearing change that un-freezes provisioning.

Persistent run history

Promote the overwritten m.status and the ephemeral output into a revisitable []run keyed by VM name, so a failed Ansible run is still readable ten minutes later. The single highest-leverage UX win.

01

Living Cockpit Hangar ⊕ Tidepool

The dashboard you never leave — a k9s-style flight deck with a real per-VM heartbeat.

Dense always-on dashboardbuild · L · cleanest incremental path

The most literal descendant of the FlameConnect reference. The frame is fixed and only the interior changes: a fleet band up top, a VM roster and a live inspector side by side, and a docked Activity log where provisioning streams in without ever taking over the screen. Moving the roster cursor re-renders the inspector live — today's viewList → enter → viewDetail swap simply disappears.

It merges two close cousins: Hangar gives the cockpit bones and the un-freezing of provisioning; Tidepool grafts the one signature that makes it feel alive — a heartbeat sparkline per running VM, fed by the guest's load average. A box with a Claude agent building shows a tall moving pulse; an idle one flatlines. Because the motion is bound to a real signal, the beauty and the monitoring are the same thing.

signature moment

You press :, the deck dims, a palette floats up. You type prov api and hit enter — and instead of the screen blanking to two minutes of Ansible, a line slides into the Activity pane and keeps climbing on its own while you shell into another box. The build came to you instead of stealing the room.

Key interactions

↑↓move cursor — inspector re-renders live
enterexpand a job into full scrolling output
n s x rcreate / start / stop / restart — streams into Activity

Pros

  • Delivers the FlameConnect silhouette mapped 1:1 onto sand's domain
  • Kills full-screen view swaps — list, detail and secrets co-visible
  • Un-freezes provisioning: manage several VMs at once
  • The heartbeat answers “which box is working, which is safe to delete?”
  • The fleet band turns “am I about to over-commit?” into a glance

Cons

  • Biggest structural cost: decoupling streaming from the viewProgress takeover
  • Dense by nature; payoff scales with fleet size
  • Flicker-free repaints and the dimmed-deck palette really want Bubble Tea v2
  • The heartbeat needs real per-guest load, which Lima doesn't expose — careful throttling or a guest agent
feasibility

A weekend gets a live master-detail deck with a fake log. The hard 60% is behavioural: refactoring beginStream into a background job registry, three-pane focus routing, the palette overlay (v2), and cursor-preserving auto-refresh. Ship the shell + single-job streaming on v1, then take v2 for the polish layer.

02

Lantern palette-first

A calm, near-empty room of VMs that you light on demand — every action one fuzzy line away.

Palette-first minimalismbuild · M–L

One radical bet: delete almost every single-key verb and route the whole app through one floating fuzzy palette (^p). The resting screen is a low-chrome list — glyph, name, a whisper of detail. Movement is arrows/jk; enter toggles a row open into an inline accordion. That's the entire chrome.

The palette composes verb and object in one fuzzy query. A > prefix is command mode, gated by the selected VM's state. >prov feat resolves both halves at once into one scoped result, so you never pick a target separately. Each result shows a right-aligned chord that teaches the shortcut as you graduate from typing to muscle memory. Help is just the palette pre-seeded with >, so it can never drift.

signature moment

Twelve near-identical clones, and you want the login sandbox re-provisioned. You press ^p, type prov feat, one line resolves both halves — “Provision → feat-login” — enter. The palette lowers, a single line appears at the foot of the still-calm list, and you immediately raise the lantern again for the next one.

Key interactions

^praise the palette (or just start typing)
>prov featname verb + object in one line
entertoggle an inline detail accordion
?opens the palette pre-seeded with ›

Pros

  • Purest expression of the crafted, uncluttered feel — five screen types collapse to one surface plus an overlay
  • Kills the keybinding crisis: verbs live in a registry, not on scarce letters
  • Adding “reprovision” later costs one registry entry, shown everywhere automatically
  • Verb+object-in-one-line is fast for similarly-named clones, and doubles as jump-to-VM

Cons

  • Cold-start discoverability: an empty screen assumes you know ^p
  • Typing-tax on repetitive work until chords are learned
  • Palette-hosted wizards are cramped — the 11-field create form is fiddly to page
  • Verb/object fuzzy composition is ambiguous at the edges; needs careful ranking
feasibility

The mechanics are easy today (single registry, fuzzy via bubbles/list's sahilm/fuzzy dep, a lighter list render, the docked chip reusing beginStream). The v2-motivated part is compositing the floating card over a dimmed background and hosting the wizards as inline palette pages. Could share the exact same engine as the Living Cockpit — the two are alternate shells over one core.

03

sandmodal modal bulk power-tool

The fleet is an editable buffer; verbs are edits. Drive dozens of clones at the speed of thought.

Modal / vim bulk power-toolbuild · L

Your fleet is a text buffer. Each VM is a line. Editing the buffer edits the machines. The only concept that treats bulk fleet operations as the native case — which is what a heavy user of disposable clones does all day. You never swap screens, you change mode.

NORMAL navigates and acts on the current VM (3x stops three downward; xip stops the whole managed/base/unrelated group). SELECT (V) turns the fleet into a multi-select — one verb applies to the whole selection with a grouped confirm. COMMAND (:) is the palette as an ex line: :g/feat-/d tears down every feature sandbox. Registers make “another box exactly like this” trivial (yy yanks its recorded config, p pastes a new clone). Marks pin the boxes you live in.

signature moment

You type :g/feat-/x and hit enter. Messages echoes “matched 3 → feat-login · feat-search · feat-payments”, each row's glyph spins, and one by one they flip ◌ Stopped as the parallel stops land — the whole feature-branch fleet drained to sleep from a five-character sentence, while you're already typing yyp.

Key interactions

3x d} xipcounts + motions on every verb

Pros

  • Bulk ops are the native case, not a bolt-on — one keystroke or one :g line
  • The command line is the palette, but strictly more powerful
  • Registers make “another box like this” trivial, leveraging recorded CreateConfig
  • Muscle memory transfers instantly from vim/helix/k9s/lazygit

Cons

  • Steep and the opposite of discoverable — which-key menu and ? are essential
  • bubbles/table has no multi-select, so the Fleet buffer is hand-rolled — the biggest build cost
  • The :ex language is real language-design work with real edge cases
  • Bulk + concurrent lifecycle is dangerous: the confirm must be bullet-proof
feasibility

The frame and spine are easy; the mode state machine is medium. The two hard parts are the hand-rolled multi-select buffer and the :ex/:g parser. Registers/marks are easy in-model maps. Ship NORMAL + SELECT + a minimal exact-token : first; add :g / registers / marks second.

04

Strata spatial drill

A file manager for your fleet: bases › clones › facets › guest files, with a live preview one level ahead.

Spatial Miller-columns drillbuild · L · the most ambitious

yazi/Finder for a Lima fleet. There is no “list screen” and “detail screen” — there is one hierarchy you walk rightward and back out leftward, and a preview that renders whatever your cursor points at, one level deeper. Base images are the roots; clones hang off them; each clone opens into facets; and the Files facet keeps drilling right into the guest filesystem.

The signature is that the drill does not stop at the VM boundary. Upload is no longer a separate wizard — you drill to the target guest directory and press u to push a host file into where your cursor already is. This collapses six of today's views into one spatial model and reuses browse.GuestLister almost as-is.

signature moment

You walk claude-base › feat-login › Files, keep pressing l, and the same motion that scanned your fleet is now walking the guest checkout — src/ › components/ › Button.tsx — with the rightmost pane live-previewing the file's head over limactl. Fleet, VM, and the file inside a running guest are one continuous surface.

Key interactions

Pros

  • One mental model: scan, inspect, secrets, transfer and provision are all the same drill
  • Dramatically better file transfer — in-place drilling with a stat/head preview
  • Grouping clones under their base makes the base→clone relationship spatially obvious
  • Master-detail is inherent, not bolted on

Cons

  • Horizontal space is the enemy: four panes want ~100 cols
  • Live preview spawns limactl per hover — needs debounced, cached, stale-dropping commands
  • Lifecycle verbs have no honest spatial home — relegated to a popup
  • The most ambitious option: close to a full rewrite of the view layer
feasibility

Genuinely easy: the frame, side-by-side panes, focused-vs-dim borders, glyph status. Massive reuse: browse.GuestLister/LocalLister already do the directory listing; beginStream already produces the streamable log. The real new work is the multi-column depth state machine, the sliding-window layout math, the cached async preview, and persisting a per-VM provision log. The honest risk is scope.

05

Harbormaster slip-board of tiles

A marina control board where every VM is a moored boat you read at a glance — and any tile zooms in place.

2D card gridbuild · M–L

A harbormaster's slip board. claude-base is the boatyard; every clone is a boat in its slip. A pinned status band, a responsive grid of VM tiles, a docked Messages strip, a footer bar. Each tile shows a status glyph+colour, a badge, a subtitle, three little gauges (cpu, mem, and a real disk used/max bar), and arch + uptime.

Arrow keys move a focus ring; the focused tile wears a bold border. Single-key verbs aim at it; enter (or z) zooms it to fill the deck as a tabbed inspector folding today's detail, secrets, progress and transfer into Overview / Provision / Secrets / Cargo. The most visually warm and glanceable of the set — it answers “what's running, what's mine, what's full” in one look.

signature moment

You press n, fill the form, and instead of the screen going dark with an Ansible dump, a fresh tile slides into an empty slip wearing a ⣾ Building badge and an in-place bar that fills as Ansible marches through its roles. You arrow away, glance back, and the tile has flipped to ● Running green. The fleet grew a boat while you watched.

Key interactions

↑↓←→ hjklmove the focus ring across tiles
enter / zzoom the tile into a tabbed inspector
s x r Sstart / stop / restart / shell the focused tile
nnew clone (or the “+ new sandbox” ghost tile)
^pstate-legal palette + jump-to-VM
/ ffilter by name · toggle managed-only

Pros

  • Delivers the crafted feel: fixed frame, live interior, a board that reflows
  • At-a-glance scanning — colour, badge and a real disk bar per tile
  • Zoom folds four of today's views into one inspector
  • Committed-vs-host capacity header is a real over-commit safety win

Cons

  • Tiles are less dense than a table — needs a compact-roster fallback and sorting
  • cpu/mem gauges imply utilization sand doesn't have; label them allocation-vs-host
  • Biggest lift is the board: a new render path plus 2D focus routing
  • Zoom/confirm overlays and flicker-free repaints want v2
feasibility

Easy today: the three-region frame, tiles as bordered blocks, glyph+colour and the disk bar, header gauges, a state-gated footer. Moderate: 2D focus routing, the breakpoint classifier, the palette, auto-refresh. Hard / v2-preferred: the zoom inspector, overlays, and multiplexing the provisioner. The existing commands are all reusable — a re-composition of the view layer, not a plumbing rewrite.

These aren't mutually exclusive. The suggested path: build the shared engine first, prototype the Living Cockpit as the anchor (closest to the reference, cleanest incremental path), keep Lantern in your pocket as the palette layer on the same engine, and treat sandmodal / Strata / Harbormaster as the more opinionated bets — pick one if its paradigm matches how you actually live in sand.
