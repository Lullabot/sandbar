# Slim the bootstrap to ansible-core and remove both collection dependencies

## Objective

Delete ~190 MB of small-file unpacking from the critical path by installing `ansible-core` (1.3 MB download / 8.2 MB installed) instead of Debian's `ansible` bundle (17.7 MB / 200.7 MB). This is only safe once the playbook stops referencing collection content, so both collection uses must be replaced first.

## Skills Required

- **ansible** — replacing `community.general.locale_gen` and `ansible.posix.authorized_key` with collection-free equivalents, keeping both idempotent.
- **go** — editing the Lima `mode: dependency` script in `internal/provision/overlay.go`.

## Acceptance Criteria

- [ ] `roles/base/tasks/main.yml` no longer uses `community.general.locale_gen`; the locale is generated by an equivalent collection-free task.
- [ ] `roles/user/tasks/main.yml` no longer uses `ansible.posix.authorized_key`; authorized keys are managed with `lineinfile` (or equivalent core module).
- [ ] `grep -rE 'community\.general|ansible\.posix' roles/ site.yml` returns **no matches** on the default path.
- [ ] The Lima dependency script in `internal/provision/overlay.go` installs `ansible-core` (NOT `ansible`), plus `rsync`, `curl`, `gnupg`, and `ca-certificates`.
- [ ] The dependency script's existing fast-exit check is updated to test for the NEW tool-set (it currently checks `command -v ansible`; `ansible-core` still provides `ansible`/`ansible-playbook`, so also verify `rsync`, `curl`, `gpg` presence) — it must not short-circuit on a stale condition and skip installing the new prerequisites.
- [ ] An opt-in profiling mode exists that installs the `ansible.posix` collection on demand (via `ansible-galaxy collection install ansible.posix`) and enables `profile_tasks`; the default path stays collection-free.
- [ ] **Idempotence proven**: running the base-phase playbook twice in a row reports `changed=0` on the second run for the replaced locale and authorized_keys tasks.
- [ ] `ansible-playbook --syntax-check site.yml` passes; `go vet ./...` and `go test ./...` are green.
- [ ] On a real base build (if a Lima host is available): `dpkg -l ansible` shows the bundle is NOT installed and `dpkg -l ansible-core` shows it IS.

Use your internal Todo tool to track these and keep on track.

## Technical Requirements

- Current dependency script (`internal/provision/overlay.go`, ~lines 27–41): `set -eux -o pipefail`, a fast-exit `if command -v ansible … && command -v rsync …; then exit 0; fi`, then `apt-get update && apt-get install -y ansible rsync`. Lima re-runs provision scripts on **every boot**, including every clone's boots — which is why the fast-exit exists and must keep working.
- `community.general.locale_gen` is used at `roles/base/tasks/main.yml` ~line 28-32, gated to the base/full phase.
- `ansible.posix.authorized_key` is used at `roles/user/tasks/main.yml` ~line 64-71. It only fires when `user_github_keys_url` is set, which is empty in the Lima flow — so it serves the non-Lima/remote-host deployment path. It must keep working **there**.
- `curl` / `gnupg` / `ca-certificates` are added to the dependency script deliberately: task 3 needs them present BEFORE the single consolidated apt install, otherwise a second apt pass is structurally unavoidable.

## Input Dependencies

- Task 1 must land first: `profile_tasks` lives in the `ansible.posix` collection vendored by the fat `ansible` package. Enabling it while that package is still installed (task 1) works immediately; after this task it becomes opt-in.

## Output Artifacts

- A collection-free playbook on the default path.
- A lean Lima dependency script that also supplies task 3's apt prerequisites.
- An opt-in profiling mode that provisions `ansible.posix` on demand.

<summary>Detailed implementation guidance</summary>

**1. Replace `community.general.locale_gen`.**

Current shape (roughly):

```yaml
- name: Generate locale
  community.general.locale_gen:
    name: "{{ base_locale }}"
    state: present
```

Replace with an idempotent pair — uncomment the locale in `/etc/locale.gen`, then run `locale-gen` only when that changed:

```yaml
- name: Ensure the locale is enabled in /etc/locale.gen
  ansible.builtin.lineinfile:
    path: /etc/locale.gen
    regexp: '^#?\s*{{ base_locale }} UTF-8'
    line: "{{ base_locale }} UTF-8"
    state: present
  register: locale_line

- name: Generate the locale
  ansible.builtin.command: locale-gen
  when: locale_line.changed
  changed_when: locale_line.changed
```

The `when: locale_line.changed` is what makes this idempotent — a second run reports `changed=0`. Do NOT run `locale-gen` unconditionally.

**2. Replace `ansible.posix.authorized_key`.**

Current shape uses the collection module with `user:` and `key:` (a URL fetch). Replace with core modules. Because the key source is a URL (`user_github_keys_url`), fetch then write:

```yaml
- name: Fetch the GitHub public keys
  ansible.builtin.get_url:
    url: "{{ user_github_keys_url }}"
    dest: "/tmp/{{ user_name }}-github-keys"
    mode: "0600"
  when: user_github_keys_url | default('') | length > 0

- name: Install the authorized keys
  ansible.builtin.lineinfile:
    path: "/home/{{ user_name }}/.ssh/authorized_keys"
    line: "{{ item }}"
    create: true
    owner: "{{ user_name }}"
    group: "{{ user_name }}"
    mode: "0600"
  loop: "{{ lookup('file', '/tmp/' ~ user_name ~ '-github-keys').splitlines() }}"
  when: user_github_keys_url | default('') | length > 0
```

Keep the same `when:` guard so the Lima flow still skips it entirely. **File mode and ownership are load-bearing** — `~/.ssh/authorized_keys` must be `0600` and owned by the user, or sshd ignores it. Also ensure the `.ssh` directory exists with mode `0700`.

**3. Lima dependency script (`internal/provision/overlay.go`).**

Change the install line from `ansible rsync` to:

```
apt-get install -y ansible-core rsync curl gnupg ca-certificates
```

Update the fast-exit guard so it does not skip when the new prerequisites are missing:

```sh
if command -v ansible-playbook >/dev/null 2>&1 \
   && command -v rsync >/dev/null 2>&1 \
   && command -v curl >/dev/null 2>&1 \
   && command -v gpg >/dev/null 2>&1; then
  exit 0
fi
```

This matters because Lima re-runs the script on every boot; a stale guard that exits early would leave `curl`/`gnupg` missing and break task 3.

**4. Profiling opt-in mode.**

Because `profile_tasks` is in `ansible.posix` (verified absent from `ansible-core`), gate it. Add an env var or provisioner flag (e.g. `SAND_PROFILE=1`) that, when set, runs `ansible-galaxy collection install ansible.posix` in the guest before the playbook and enables the callback. When unset, the default path installs no collections. Document the flag (task 12 covers docs).

**5. Verify.**

- `grep -rE 'community\.general|ansible\.posix' roles/ site.yml` → no matches.
- `ansible-playbook --syntax-check site.yml`.
- Idempotence: run the base playbook twice; second run must show `changed=0` for these tasks.
